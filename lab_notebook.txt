## GAPS AND REQUESTS

### 1. GPU MEMORY (Critical blocker)
Process 998802 (llama-server) is holding ~19GB of GPU VRAM. This leaves ~4GB for training, causing repeated OOM crashes on any experiment that uses more models or larger architectures.

**Request:** If possible, run autoresearch when the llama-server is not occupying the GPU, or reduce its context size / offload to CPU. Every OOM wastes 15-30 minutes and blocks experimentation.

### 2. More VRAM for larger models
The dual CheMeleon ensemble already pushes memory limits. Adding a third model (frozen variant, latent features, etc.) or wider Chemprop would help but is blocked by memory.

**Request:** A GPU with 24GB+ free (not 4GB) would unlock structural improvements.

### 3. Additional pre-training data
The ~4139 training compounds are the bottleneck. No architecture change helps much with limited data.

**Request:** Any additional PXR data (even noisy/low-confidence) would help. ChEMBL PXR data, literature-curated datasets, or data from other assay platforms.

### 4. Cross-validation scheme
Currently using a single 80/20 train-val split. Results have ~0.01 noise from split-dependent variance.

**Request:** If the test set is truly held out, consider providing multiple train/val/test splits or stratified k-fold to get more reliable signals for optimization.

### 5. Feature access
Osmordred features exist but are underutilized (only by Ridge). The MPNN latent fingerprint extraction requires batch-level access that's awkward with current API.

**Request:** Pre-computed MPNN latent features for train/test would let XGBoost use them without GPU memory concerns.
